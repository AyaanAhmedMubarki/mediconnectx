📋 Project Overview

What is MediConnectX?

MediConnectX is a comprehensive digital healthcare platform that connects patients, doctors, and administrators in one unified system. It enables doctors to manage appointments (online and offline), patients to book consultations, and admins to oversee the entire platform.

Key Objectives

  • Reduce Healthcare Friction: Eliminate the need for phone calls and manual booking
  • 24/7 Accessibility: Patients can book appointments anytime, anywhere
  • AI-Powered Support: Built-in chatbot for health guidance
  • Revenue Tracking: Automatic payment and earnings management
  • Event Management: Health camps and wellness events
  • Dual Booking System: Both online (video) and offline (clinic) consultations

Who Uses It?

👨‍⚕️ Doctors

Manage availability, accept bookings, conduct video consultations, track earnings, view patient details

👨‍💼 Patients

Book online/offline appointments, view medical records, chat with AI assistant, join video calls

⚙️ Admins

Oversee doctors & patients, manage events, view platform statistics, generate reports

Authentication & Security

JWT Token-Based Authentication

When users login, the system generates a JWT token that proves who they are. This token is sent with every request to verify the user's identity.

Login Flow: User enters email/password ↓ Backend validates credentials ↓ Backend generates JWT token ↓ Token sent to frontend & stored in localStorage ↓ Every API request includes token in Authorization header ↓ Backend verifies token is valid before processing request

Role-Based Access Control (RBAC)

Three roles with different permissions:

  • PATIENT: Can book appointments, view own records, chat with AI
  • DOCTOR: Can create availability, view bookings, conduct consultations
  • ADMIN: Can view all data, manage users, oversee events

🗄️ Database Design

Core Entities (Tables)

Entity Purpose Key Fields
User All system users id, email, password (hashed), name, role, phone
Doctor Doctor-specific info id, specialization, experience, bio, photo, user (reference)
Patient Patient-specific info id, medicalHistory, healthMetrics, photo, user (reference)
DoctorLocation Clinic locations id, doctor, clinicName, address, city, state, pincode, phone
OfflineSlot Clinic appointment slots id, doctor, location, slotDate, startTime, endTime, status, patient, createdAt
Meet Online consultations id, doctor, patient, appointmentDate, startTime, endTime, meetLink, status
Event Health events/camps id, name, description, eventDate, location, capacity, registrations
EventRegistration Event participant records id, user, event, registrationDate, status

Entity Relationships

User (1) ─────────── (Many) Doctor │ └─ (1) ─────────── (Many) Patient │ └─ (1) ─────────── (Many) EventRegistration Doctor (1) ─────────── (Many) DoctorLocation Doctor (1) ─────────── (Many) OfflineSlot Doctor (1) ─────────── (Many) Meet DoctorLocation (1) ─────────── (Many) OfflineSlot Patient (1) ─────────── (Many) OfflineSlot (as booker) Patient (1) ─────────── (Many) Meet (as consultant) Patient (1) ─────────── (Many) EventRegistration Event (1) ─────────── (Many) EventRegistration

Key Database Concepts

  • Foreign Keys: Create relationships between tables (e.g., Doctor table references User table)
  • Eager Fetching: When fetching a Doctor, automatically load their User info and Locations
  • Timestamps: createdAt, updatedAt fields track when records are created/modified
  • Enums: Slot status is OPEN or BOOKED (not stored as text)
  • Indexing: Frequently searched fields (email, specialization) are indexed for faster queries

📡 REST API Endpoints

Authentication

POST /api/auth/register

Create new user account (patient or doctor)

POST /api/auth/login

User login - returns JWT token

Doctor Endpoints

GET /api/doctor

Get all doctors (for patient searching)

GET /api/doctor/{doctorId}

Get specific doctor details

PUT /api/doctor/{doctorId}

Update doctor profile (name, bio, photo)

POST /api/doctor/{doctorId}/location

Add clinic location

GET /api/doctor/{doctorId}/location

Get doctor's all clinic locations

Online Consultations (Meets)

POST /api/meets

Doctor books a meet slot (creates availability)

GET /api/meets/doctor/{doctorId}

Get all meets for a doctor

GET /api/meets/patient/{patientId}

Get all meets for a patient

POST /api/meets/{meetId}/book

Patient books a meet with doctor

GET /api/meets/{meetId}/available

Get available meets for specific doctor and date

Offline Consultations (Slots)

POST /api/offline-slots/create

Doctor creates offline availability slots

GET /api/offline-slots

Get all offline slots (admin view)

GET /api/offline-slots/doctor/{doctorId}

Get doctor's offline consultations

GET /api/offline-slots/patient/{patientId}

Get patient's offline booking history

POST /api/offline-slots/{slotId}/book

Patient books an offline slot

GET /api/offline-slots/available

Get all available slots for filtering

Events

POST /api/events

Admin creates new health event

GET /api/events

Get all events

POST /api/events/{eventId}/register

Patient registers for event

Request Format Example

{ "email": "patient@example.com", "password": "SecurePass123!", "name": "John Doe", "role": "PATIENT" }

Response Format Example

{ "status": "SUCCESS", "message": "Login successful", "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...", "user": { "id": 1, "email": "patient@example.com", "name": "John Doe", "role": "PATIENT" } }

🔄 How It All Works Together

User Registration & Login Flow

User fills registration form ↓ Frontend sends email/password/name/role to /api/auth/register ↓ Backend validates input (not empty, email format valid) ↓ Backend checks if email already exists in database ↓ Backend hashes password using bcrypt (for security) ↓ Backend creates User record in database ↓ If role is DOCTOR or PATIENT, creates Doctor/Patient record ↓ Response sent to frontend with success message

Patient Booking Online Consultation

Patient opens PatientMeets.html ↓ Page loads all doctors from /api/doctor endpoint ↓ Patient selects doctor and dates using filters ↓ Page calls /api/meets/doctor/{doctorId}/available ↓ Backend returns available meeting slots for that doctor ↓ Patient sees slots with doctor name, date, time ↓ Patient clicks "Book Appointment" ↓ Frontend sends POST to /api/meets/{meetId}/book with patientId ↓ Backend checks if slot is still available ↓ Backend sets patientId on the Meet record ↓ Backend changes Meet status from AVAILABLE to BOOKED ↓ Email sent to patient and doctor about booking ↓ Frontend shows "Booking Successful"

Doctor Creating Offline Clinic Slots

Doctor opens DoctorAvailability.html ↓ Page shows list of doctor's clinic locations ↓ Doctor selects a location ↓ Doctor picks a date, start time, end time ↓ If time is 2 hours (e.g., 5:00 PM - 7:00 PM): - Automatically creates 4 slots of 30 minutes each - Slot 1: 5:00 - 5:30 PM - Slot 2: 5:30 - 6:00 PM - Slot 3: 6:00 - 6:30 PM - Slot 4: 6:30 - 7:00 PM ↓ Frontend sends POST to /api/offline-slots/create ↓ Backend validates all 4 slots don't overlap with existing slots ↓ Backend creates all 4 OfflineSlot records with status=OPEN ↓ Frontend shows "Slots created successfully"

Patient Booking Offline Clinic Appointment

Patient opens PatientOfflineBooking.html ↓ Page loads all available offline slots from /api/offline-slots/available ↓ Slots are grouped by doctor and clinic location ↓ Patient sees doctor cards with clinic details ↓ Patient selects a doctor to see available time slots ↓ Page filters slots by date if patient selects a date ↓ Patient clicks on a specific slot (e.g., 5:30 - 6:00 PM) ↓ Frontend sends POST to /api/offline-slots/{slotId}/book ↓ Backend sets patientId on OfflineSlot record ↓ Backend changes slot status from OPEN to BOOKED ↓ Email sent to patient with slot details and clinic info ↓ Patient can see booking in "My Offline Consultations"

Admin Viewing Platform Data

Admin opens AdminDashboard.html ↓ Page calls multiple API endpoints in parallel: - /api/doctor → count total doctors - /api/patient → count total patients - /api/events → count total events - /api/event-registrations → count registrations ↓ Backend returns counts ↓ Page displays statistics in cards ↓ Admin clicks "All Online Appointments" ↓ Opens AdminMeets.html and loads all meets ↓ Admin can filter by status, doctor, date ↓ Admin clicks "All Offline Appointments" ↓ Opens AdminOfflineConsults.html and loads all slots ↓ Admin can see status (Booked/Completed/Open) ↓ Admin can view doctor name, patient name, clinic info

AI Chatbot Conversation

User clicks "AI Health Chatbot" card on home page ↓ Modal dialog opens with chat interface ↓ User types question and presses Enter ↓ Frontend creates chatHistory array (max 20 messages) ↓ Frontend sends POST to OpenAI API with: - System prompt (role-specific: doctor/patient/admin) - Chat history (previous messages for context) - User's new message - API key in Authorization header ↓ OpenAI processes request and returns response ↓ Response displayed in chat as "bot" message ↓ Previous 20 messages kept for context ↓ Older messages discarded (to save tokens and costs)

Status Determination Logic

How System Knows if Appointment is "Completed" or "Booked"

Current Time: 2026-06-17 10:00 AM Appointment Details: - Date: 2026-06-17 - Start Time: 09:00 AM - End Time: 10:30 AM Calculation: Appointment End DateTime = 2026-06-17 10:30 AM Current Time = 2026-06-17 10:00 AM If (appointmentEndDateTime <= currentTime): Status = "COMPLETED" (show blue badge) Else: Status = "BOOKED" (show green badge) Result: 10:30 AM > 10:00 AM → Still ongoing → "BOOKED"

Prerequisites

  • Maven installed (for dependency management)
  • MySQL 8.0 installed and running

Database Setup

-- Create database CREATE DATABASE mediconnectx; -- All tables are created automatically by Hibernate -- (Spring Boot does this on first run)

spring.application.name=mediconnectx spring.jpa.hibernate.ddl-auto=update spring.jpa.show-sql=false spring.jpa.properties.hibernate.dialect=org.hibernate.dialect.MySQL8Dialect spring.jpa.properties.hibernate.jdbc.batch_size=10 # Database Connection spring.datasource.url=jdbc:mysql://localhost:3306/mediconnectx spring.datasource.username=root spring.datasource.password=yourpassword spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver # JWT Configuration security.jwt.secret=YourSecretKeyHere123!@# security.jwt.expiration=86400000 # Server Port server.port=8080
